# update-rules

Edit the rule layer with the correct scope, then sync. The dotfiles repo is the source of truth for shared AI rules and MCP definitions; this skill picks the right file to touch and runs the fan-out.

## Quick decision tree

Ask first: **which tools should obey this rule?**

| Scope | Edit | Affects |
|---|---|---|
| All AI tools (Claude + Codex + Gemini + OpenCode) | `~/.dotfiles/.config/rulesync-global/.rulesync/rules/overview.md` + run sync | Everything via rulesync fan-out |
| Claude-only (project / user-global) | `~/.dotfiles/.claude/CLAUDE.md` | Claude Code (stow-symlinked to `~/.claude/CLAUDE.md`) |
| Codex-only (repo root) | `~/.dotfiles/AGENTS.md` | Codex CLI (auto-loaded from repo root) |
| One project | `<project>/CLAUDE.md` or `<project>/.claude/CLAUDE.md` and/or `<project>/AGENTS.md` | Only that repo |
| MCP server (cross-tool) | `~/.dotfiles/.config/rulesync-global/.rulesync/mcp.json` + run sync | Claude / Codex / Gemini / OpenCode MCP configs |

If the user just says "add this rule" without specifying scope, **ask** before editing — the wrong layer creates drift.

## Files and what they own

```
~/.dotfiles/
├── AGENTS.md                                              # Codex root (auto-loaded by Codex from repo root)
├── .claude/CLAUDE.md                                      # Claude project rules → stowed to ~/.claude/CLAUDE.md
└── .config/
    ├── rulesync-global/
    │   └── .rulesync/
    │       ├── rules/overview.md                          # CANONICAL shared rules (front-matter: root, targets: ["*"])
    │       └── mcp.json                                   # Shared MCP server definitions
    └── codex/
        └── sync-ai-global-config.sh                       # Runs `rulesync generate` then merges into ~/.codex, ~/.gemini, ~/.config/opencode
```

What the sync script writes (and explicitly does NOT write):

- Writes: `~/.codex/AGENTS.md`, `~/.codex/config.toml` (managed block), `~/.gemini/GEMINI.md`, `~/.gemini/settings.json`, `~/.config/opencode/AGENTS.md`, `~/.config/opencode/opencode.jsonc`
- Does NOT write: `~/.claude/CLAUDE.md` or `~/.claude/.mcp.json` — those are stow-managed from `~/.dotfiles/.claude/`. Editing the dotfiles file is enough.

## Workflows

### Add a rule that applies to every AI tool

1. Open `~/.dotfiles/.config/rulesync-global/.rulesync/rules/overview.md`.
2. Append to the appropriate section (Workflow Expectations, Verification, Infrastructure Questions, Prefer skills, Agent Delegation, Project Mapping, Compact Handoff). Keep it terse — one bullet, with rationale only when non-obvious.
3. Run sync:
   ```bash
   ~/.dotfiles/.config/codex/sync-ai-global-config.sh
   ```
4. If the rule is also worth pinning in the Claude-flavored `CLAUDE.md` (richer prose, eval format, memory routing), mirror it there. The Claude file is intentionally a superset, not a copy.
5. Commit with a message like `rulesync: add <one-line summary>`.

### Add a Claude-only rule (eval, lessons, plan-mode behavior)

1. Edit `~/.dotfiles/.claude/CLAUDE.md`. The change is live for the next Claude session — no sync needed (stow symlink → `~/.claude/CLAUDE.md`).
2. If the rule is general enough to belong in the shared layer, prefer the rulesync overview path above instead.

### Add a Codex-only rule

1. Edit `~/.dotfiles/AGENTS.md`. No sync runs against this file — it's the Codex root. The sync only writes `~/.codex/AGENTS.md` (a different copy generated by rulesync from the shared overview).
2. If the rule should also reach Codex's home dir, put it in the rulesync overview instead.

### Add a project-specific rule

1. Use `<project>/CLAUDE.md` (Claude) or `<project>/AGENTS.md` (Codex / others). Don't put project-only rules in the dotfiles layer.
2. For project-specific *corrections* (the kind that come from a user redirect), use `~/.agent/lessons/{project}.md` instead — the SessionStart hook auto-injects the last 20 lines.

### Register a new MCP server

1. Edit `~/.dotfiles/.config/rulesync-global/.rulesync/mcp.json` (top-level `mcpServers` map).
2. Run sync. The script merges the entries into each tool's native MCP config.
3. Restart any tool already running (Claude Code, Codex) to pick up the new server.

### Register a new skill in the CLAUDE.md skills list

The "Prefer skills over raw tooling and MCPs" section in `~/.dotfiles/.claude/CLAUDE.md` should mirror `ls ~/.dotfiles/.claude/skills/`. After creating a new skill:

```bash
# Show drift between the on-disk skills and what CLAUDE.md mentions:
diff <(ls ~/.dotfiles/.claude/skills/ | sort) \
     <(grep -oP '`[a-z][a-z0-9-]+`' ~/.dotfiles/.claude/CLAUDE.md | tr -d '`' | sort -u)
```

Add a one-line bullet under the right category (Infra/ops, Notes/memory, Workstreams, Authoring/config) with the same `description:` style used by neighbors.

### Remove a rule

1. Locate it: it lives in exactly one of the four files above (or a project repo). `grep -nF "<keyword>" ~/.dotfiles/.claude/CLAUDE.md ~/.dotfiles/AGENTS.md ~/.dotfiles/.config/rulesync-global/.rulesync/rules/overview.md`.
2. Remove the bullet, leave neighboring sections untouched.
3. Run sync if the file was the rulesync overview or `mcp.json`.

### Audit drift between layers

```bash
# What's in the shared overview vs the Claude-flavored superset?
diff -u ~/.dotfiles/.config/rulesync-global/.rulesync/rules/overview.md \
        ~/.dotfiles/.claude/CLAUDE.md | less

# What's the Codex root saying that the shared overview isn't?
diff -u ~/.dotfiles/AGENTS.md \
        ~/.dotfiles/.config/rulesync-global/.rulesync/rules/overview.md | less

# Show what rulesync would generate without writing it (dry run via stage dir):
cd "$(mktemp -d)" && cp -R ~/.dotfiles/.config/rulesync-global/. .
rulesync generate --targets codexcli,geminicli,opencode --features rules,mcp
ls -la AGENTS.md GEMINI.md .codex/ .gemini/ opencode.jsonc
```

## Style rules for new bullets

- Lead with the imperative or the fact, not throat-clearing.
- Add a `Why:` clause only when the rationale isn't obvious from the rule itself.
- Don't reference today's task / commit / PR — rules outlive incidents.
- Don't duplicate across layers. If a rule lives in the shared overview, it doesn't also need to be in the Claude superset unless the Claude-side context (eval format, memory routing) materially changes how to apply it.

## What this skill does NOT do

- Settings, permissions, hooks, env vars → `update-config` skill.
- Per-correction lessons → write directly to `~/.agent/lessons/{project}.md`. The SessionStart hook injects them; no rule-layer change needed.
- User preferences that should ride across projects → `mem0-ops` skill (mem0 at `mem0.example.internal`), not the rule layer.
- Project runbooks → live in the project repo's docs, not in CLAUDE.md or AGENTS.md.

## Verification after a change

- For shared-layer edits: `~/.dotfiles/.config/codex/sync-ai-global-config.sh` runs to completion (exit 0) and modifies the expected files in `~/.codex`, `~/.gemini`, `~/.config/opencode`. Check `git diff` in those dirs.
- For Claude-only edits: open a new Claude session and verify the rule is in the `<system-reminder>` CLAUDE.md block. (Stow symlink means the file at `~/.claude/CLAUDE.md` should already match.)
- For MCP additions: list MCP servers in the target tool (e.g., `claude mcp list` or `:McpList` in Codex) and confirm the new server is enumerated.
