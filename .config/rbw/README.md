# rbw — Bitwarden CLI → self-hosted Vaultwarden

[`rbw`](https://github.com/doy/rbw) is a lightweight Rust Bitwarden client. We
point it at the self-hosted Vaultwarden at **`vault.kblab.me`** so secrets
(Cloudflare API tokens, etc.) can be sourced at shell-init with `rbw get`
instead of living as stale literals in `~/.bash_profile`.

## Provisioning

- **Package:** `rbw` is in `PACKAGES_DEV_ARCH` / `PACKAGES_DEV_MAC`
  (`installation_scripts/packages.conf`). Arch pulls it from the `extra` repo;
  macOS from Homebrew.
- **Config:** `setup_rbw()` in `installation_scripts/linux/install_arch.sh` runs
  `rbw config set` for `base_url`, `email`, and `lock_timeout` (idempotent).
- **Login is manual** — `rbw login` needs the Vaultwarden master password
  interactively, and the session/device state it writes is ephemeral auth-state
  we never script (per the Auth-State Safety rule). The installer surfaces the
  command; you run it once per machine.

`config.json` is **not** stowed/git-tracked: rbw writes a per-machine
`device_id` into it at runtime, so it's machine-local state, regenerated by
`setup_rbw()` on each box.

## First-time setup on a new machine

```bash
# 1. installed + configured by the installer; or by hand:
rbw config set base_url https://vault.kblab.me
rbw config set email    kblack0610@gmail.com

# 2. log in (prompts for the Vaultwarden master password)
rbw login

# 3. unlock for the session
rbw unlock
```

## Storing & sourcing the Cloudflare tokens

```bash
# store (one-time; opens $EDITOR for the secret body)
rbw add 'Cloudflare API Token'          # DNS:Edit + Tunnel:Edit, 40-char API token

# read
rbw get 'Cloudflare API Token'
```

Then in `~/.bash_profile` (you edit this file — it is auth-state, never
automated), replace the stale literal:

```bash
# before
export CLOUDFLARE_API_TOKEN=<stale-literal>

# after — single source of truth, survives rotation
export CLOUDFLARE_API_TOKEN=$(rbw get 'Cloudflare API Token' 2>/dev/null)
```

> **Honest scope:** Vaultwarden is the *storage/distribution* layer, not an
> auto-rotator. Cloudflare tokens are still rotated by hand in the CF dashboard;
> the win is that rotation becomes a single Vaultwarden update instead of
> editing literals on every machine. The cluster's DNS token stays on SOPS
> (GitOps, Flux-decrypted) — optionally mirror its value here as the
> human-canonical copy.

## Cloudflare token gotcha (why this exists)

A real Cloudflare **API token** used as `Authorization: Bearer …` is exactly
**40 characters**, alphanumeric. If `…/user/tokens/verify` returns
`Invalid API Token`, check the length/shape first — a `wcfat_`-prefixed or
longer value is a *different credential type* (not the API token secret), and
no amount of correct dashboard permissions will make it authenticate against
`api.cloudflare.com/client/v4`. CF only reveals the 40-char secret **once**, at
create/roll time — store that value here immediately.
